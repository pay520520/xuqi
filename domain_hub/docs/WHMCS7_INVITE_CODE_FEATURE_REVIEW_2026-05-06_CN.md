# WHMCS7 域名分发插件前后端功能与“邀请注册邀请码”改造评估（2026-05-06）

## 1. 检查范围与方法
- 代码范围：`domain_hub/` 目录下 PHP 服务层、控制器层、模板层与前端脚本。
- 检查重点：
  - 邀请注册（Invite Registration）现有规则与前后端一致性。
  - 根域名邀请码（Rootdomain Invite）与邀请注册（Invite Registration）功能边界。
  - 你提出的需求：**手动生成 + 批量数量输入 + 每页最多5条 + 已使用不显示** 的可行性与改造复杂度。

## 2. 现状结论（与你描述是否一致）

### 2.1 “邀请注册-我的邀请码”当前确实是单码模式，且被使用后自动轮换
证据：
1. 前端展示只有单一输入框（“我的邀请码”），并提示“邀请码仅限一次使用，被使用后会自动生成新的邀请码”。
2. 后端在邀请码被成功使用后，会执行 `rotateCode()`（自动轮换）。

这与你描述的“默认一个，用掉再下一个”一致。

### 2.2 目前并非“码池/多码库存”模型
- `mod_cloudflare_invite_registration_unlock`（由服务常量 `TABLE_UNLOCK` 表示）按用户维护单条 profile + 当前邀请码。
- 邀请记录在日志/claims中，不是“多条待用邀请码库存”。

因此你要的“手动输入数量一次生成多条并分页管理未使用码”属于**功能模型升级**，不是简单UI改动。

## 3. 与需求差距（Gap）
目标需求：
1. 用户手动点击生成，不再自动补码。
2. 用户可输入一次生成数量（且不超过可生成上限）。
3. 生成后分页显示（每页最多5个）。
4. 已使用邀请码不在“我的邀请码”显示。

当前实现差距：
- 数据模型：缺少“邀请码库存表”与“状态字段（unused/used）”。
- 业务流程：当前是“单码 + 使用后立即轮换”，不支持批量生成。
- 前端交互：当前只展示一个 code 字段，无“生成数量输入”“生成按钮”“邀请码列表分页”。
- 安全/并发：批量生成需要原子扣减剩余额度、防重与并发控制。

## 4. 推荐方案（可落地）

## 4.1 方案A（推荐）：新增邀请码池表，保留旧逻辑兼容迁移

### 新增数据表（示例）
`mod_cloudflare_invite_code_pool`
- `id`
- `owner_userid`
- `invite_code`（唯一）
- `status`（unused/used/disabled）
- `used_by_userid`（nullable）
- `used_at`（nullable）
- `batch_no`（可选，便于审计）
- `created_at` / `updated_at`

### 配套改造
1. **生成接口**：`generateInviteCodes($userId, $count)`
   - 入参校验：`count >=1` 且 `count <= remaining`。
   - 事务内处理：锁用户配额 -> 批量生成唯一 code -> 写入 pool。
2. **解锁接口**（用户使用邀请码）
   - 在 pool 中按 code 查 `unused`，命中后置 `used`，写 usage log。
   - 禁止使用自己的 code，保留原封禁与资格校验。
3. **列表接口**
   - 仅查询 `status=unused`。
   - 每页固定 `perPage=5`。
4. **前端页面**
   - 新增“生成数量 input + 生成按钮”。
   - 下方表格展示 unused codes + 分页。

### 兼容策略
- 迁移期可读取旧 `TABLE_UNLOCK.invite_code` 作为“历史单码来源”；新生成走 pool。
- 当 old code 被使用后，逐步引导进入 pool 模式。

## 4.2 方案B（不推荐）：在现有单码表上硬改为多码字段
- 会导致字段语义混乱、查询效率与可维护性差。
- 后续风控审计困难。

## 5. 难度评估
- **纯前端改动**：低（但无法实现目标）。
- **端到端完整实现（推荐方案A）**：中-高。
  - 预计工作量：
    - DB migration：0.5 天
    - 服务层改造与事务/并发处理：1~2 天
    - 前端模板+交互+分页：1 天
    - 联调与回归：1 天
  - 合计：约 3~5 个工作日（取决于测试深度与历史数据兼容要求）。

## 6. 风险点与控制
1. **并发超发**：必须事务+行级锁（或等效手段）保证 remaining 计算一致。
2. **唯一冲突**：生成code需唯一索引并重试。
3. **权限越权**：列表/生成接口强制绑定当前登录 userId。
4. **旧数据兼容**：需要明确历史邀请码的去向与可见性策略。
5. **风控一致性**：保持原有封禁、最小月龄、邀请上限策略。

## 7. 建议实施顺序（最稳妥）
1. 上线 DB 新表 + 只读能力。
2. 上线生成接口（后台开关灰度）。
3. 前端“手动生成 + 每页5条未使用列表”。
4. 最后切换解锁消费逻辑到 pool。
5. 稳定后下线自动轮换逻辑。

## 8. 结论
- 你的需求是**可以实现**的。
- 但当前代码是“单码自动轮换”架构，需做一次中等规模重构。
- 推荐采用“邀请码池表 + 事务控制 + 前端分页”的方案，兼容与可维护性最佳。
